1. Readability and Naming ConventionsChoose descriptive names: Variables, functions, and classes should reveal their intent. Use pronounceable and searchable names so others can understand what a variable holds without guessing.Avoid cleverness: Write code for humans first, machines second. Clear, explicit code is always better than dense, cryptic one-liners.2. Functions and MethodsSmall and focused: Functions should be small and do only one thing. If a function tries to handle multiple tasks, split it into smaller, independently named helper functions.Few arguments: Try to minimize the number of arguments a function takes. If a function requires more than three arguments, it often means it is trying to do too much.Avoid side effects: Functions should only do exactly what their name states and not cause hidden changes (side effects) elsewhere in the system.3. Core Software Principles (DRY, KISS, YAGNI)DRY (Don't Repeat Yourself): Avoid duplicating code. Extract common functionality into a single piece of reusable code so that updates only need to be made in one place.KISS (Keep It Simple, Stupid): Keep your solutions as simple as possible. Avoid overcomplicating or prematurely engineering your code for edge cases that don't exist yet.YAGNI (You Ain't Gonna Need It): Only write the code necessary to solve the exact problem you are currently facing.4. Abstraction and FlowSLAP (Single Level of Abstraction Principle): Keep each block of code focused on a single layer of detail. A high-level workflow should not be bogged down with low-level details like calculating specific arrays or formatting timestamps.Avoid excessive nesting: Deeply nested if statements become very difficult to read. Flatten your code by checking for disqualifying conditions and exiting early (using "guard clauses" or early returns).5. Comments and OrganizationCode should explain itself: Always try to explain yourself in the code rather than relying on comments.Comment the "Why," not the "What": If you must use a comment, it should explain the intent, consequence, or business rule behind a block of code, not simply describe what a line of code does.Formatting and structure: Keep related concepts and variables declared physically close to where they are used. Use whitespace to create visual space between unrelated concepts.6. SOLID PrinciplesFor larger structural systems, applying object-oriented design principles helps keep code modular and manageable:Single Responsibility Principle: A class or module should have only one reason to change.Open/Closed Principle: Software entities should be open for extension but closed for modification.Liskov Substitution Principle: Objects of a superclass should be replaceable with objects of its subclasses without breaking the application.Interface Segregation Principle: Many client-specific interfaces are better than one general-purpose interface.Dependency Inversion Principle: Depend on abstractions, not on concrete implementations.
